This project covers the complete Business Intelligence development lifecycle for a fictional retail company “Mechman Retail”. The dataset spans 228,265 sales transactions across May 2017 to October 2020, covering $172M in revenue at an average profit margin of 49.8% across 403 customers, 672 products, 8 salespeople, and 9 US sales territories.
Phase 1 — SQL Server The project implements a three-tier architecture across MechmanRetail_SOURCE, MechmanRetail_DEV, and MechmanRetail_PROD. SOURCE holds the full raw dataset and represents the operational source system. DEV is a staging environment that holds only the latest unprocessed month at any one time. PROD holds the full historical analytical dataset that the published Power BI report connects to. The source table dbo.Sale was left unchanged with its original spaced column naming convention. Column standardisation was applied entirely at the view layer — vw_Sales_Base exposes only the 12 columns actively consumed by the Power BI model with clean camelCase aliases and WITH SCHEMABINDING, making it the single standardisation point in the project. Five unused columns were deliberately excluded to keep the semantic model lean. vw_Salesperson_KPI pre-aggregates annual salesperson performance using a RANK() window function partitioned by InvoiceYear — retained as a database-layer analytical endpoint for direct SQL access, but superseded in the published report by a RANKX DAX measure that responds dynamically to filter context. vw_Territory_Sales exposes the distinct CityKey, SalespersonKey, and CustomerKey combinations to support the SQL Server RLS predicate function — retained as a defence-in-depth measure alongside the RLS policy, both of which are inactive by design since Power BI Import mode requires RLS to be enforced at the Power BI layer. Three stored procedures implement the incremental pipeline — sp_LoadLatestMonthToDEV detects the latest month in SOURCE not yet present in PROD at month-level granularity and stages it into DEV automatically, sp_EnvironmentHealthCheck provides read-only diagnostic validation returning detailed counts and a PASS/FAIL summary for manual inspection, and sp_PromoteDEVtoPROD validates DEV inline, detects new months, promotes only those rows into PROD inside a transaction with row count validation, then validates PROD inline after promotion — preserving all historical data with no manual intervention required. Nine database users were created WITHOUT LOGIN — one per sales territory manager plus BI_Admin — with SELECT granted on vw_Sales_Base only. A SQL Server RLS predicate function and security policy were created with STATE = OFF, documented as a defence-in-depth measure for direct database access scenarios.
Phase 2 — Power Query Nine queries were built in Power Query Editor transforming six source files — SQL Server, Excel, CSV, and Folder — into a clean star schema model. Key decisions include a galaxy-to-star schema flattening of the geography dimension by merging City.csv and StateProvince.csv via a Left Outer join, a folder connection for the 2017–2019 target CSVs with BuyingGroup dirty data cleaned using Trim and Capitalize Each Word, a custom M code date construction for the 2020 targets sheet converting month names to end-of-month dates, and an Append producing dim_Targets_All with EndOfMonth and BuyingGroup as the natural composite key. The dim_Employee table retains Email and ParentEmployeeKey specifically to support hierarchy-aware RLS in Power BI.
Phase 3 — Power BI Desktop The data model uses eight relationships — five active in a star schema pattern with Single cross-filter direction, and three inactive covering the role-playing DeliveryDateKey dimension and the dim_Targets_All connections activated via USERELATIONSHIP() inside DAX measures. A Time Intelligence calculation group with seven items — Actual, MTD, QTD, YTD, LY, YoY %, Rolling 12M — applies to any additive measure placed in a visual via SELECTEDMEASURE(), replacing what would otherwise be twelve individually written time intelligence measures. A field parameter enables switching between Total Revenue and Total Profit across visuals on Pages 1 and 3. Twelve standalone DAX measures handle ratios, targets, shipping efficiency, and rankings — including Avg Days to Ship using AVERAGEX with DATEDIFF, Target Revenue activating both inactive dim_Targets_All relationships simultaneously via USERELATIONSHIP(), and Salesperson Revenue Rank using RANKX with DENSE ranking and HASONEVALUE protection. Row Level Security uses USERPRINCIPALNAME() matched against dim_Employee[Email] with PATH() and PATHCONTAINS() traversing ParentEmployeeKey for hierarchy-aware manager access. The four-page report covers Executive Summary, Sales Performance, Product Intelligence, and Territory and Customers, with a hidden City Detail drill through page accessible by right-clicking any StateProvince data point. A MonthStart calculated column using DATE() at monthly grain enables the forecast visual to operate on a continuous date axis with readable MMM YYYY labels.
