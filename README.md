This project covers the complete Business Intelligence development lifecycle for a fictional retail company — Mechman Retail — built on the Wide World Importers dataset. The dataset spans 228,265 sales transactions across May 2017 to October 2020, covering $172M in revenue at an average profit margin of 49.8% across 403 customers, 672 products, 8 salespeople, and 9 US sales territories.
Phase 1 — SQL Server The project implements a three-tier architecture across MechmanRetail_SOURCE, MechmanRetail_DEV, and MechmanRetail_PROD. SOURCE holds the full raw dataset. DEV is a staging environment holding only the latest unprocessed month. PROD holds the full historical dataset that Power BI connects to. The source table dbo.Sale was left unchanged — column standardisation applied at the view layer via vw_Sales_Base which exposes only the 12 consumed columns with clean camelCase aliases and WITH SCHEMABINDING. Five unused columns deliberately excluded. vw_Salesperson_KPI retained as a database-layer analytical endpoint using RANK() window function. vw_Territory_Sales retained as RLS bridge alongside the inactive policy. Three stored procedures implement the incremental pipeline — sp_LoadLatestMonthToDEV detects the latest month in SOURCE absent from PROD at month-level granularity and stages it into DEV automatically, sp_EnvironmentHealthCheck provides read-only diagnostic validation, and sp_PromoteDEVtoPROD validates DEV inline, promotes only new months inside a transaction with row count validation, then validates PROD inline. Nine WITHOUT LOGIN users created with SELECT on vw_Sales_Base only. RLS policy STATE = OFF — defence-in-depth measure for direct database access scenarios.
Phase 2 — Power Query Nine queries built transforming six source files into a clean star schema. Key decisions: galaxy-to-star schema flattening of geography via Left Outer join on StateProvinceKey, folder connection for 2017–2019 target CSVs with BuyingGroup dirty data cleaned using Trim and Capitalize Each Word, M code date construction for 2020 targets converting month names to end-of-month dates, dim_Targets_All with EndOfMonth and BuyingGroup as natural composite key. dim_Employee retains Email and ParentEmployeeKey for hierarchy-aware RLS. MonthStart column added to dim_Date using DATE() at monthly grain for forecast visual X axis.
Phase 3 — Power BI Desktop Eight relationships — five active star schema with Single cross-filter, three inactive activated via USERELATIONSHIP(). Field parameter Metric switches between Total Revenue and Total Profit on Pages 1 and 3. 25 standalone DAX measures — time intelligence written individually for both Revenue and Profit rather than through a calculation group, allowing each measure to be placed deliberately into specific visuals without page-wide slicer complications. Key measures: Avg Days to Ship using AVERAGEX with DATEDIFF, Target Revenue activating both inactive dim_Targets_All relationships simultaneously, Salesperson Revenue Rank using RANKX with DENSE and HASONEVALUE. RLS uses USERPRINCIPALNAME() matched against dim_Employee[Email] with PATH() and PATHCONTAINS() for hierarchy-aware manager access. Four-page report with hidden City Detail drill through page. Forecast visual using MonthStart X axis.
